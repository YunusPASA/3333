1 Project Overview

In this project, you will implement a GNU assembly language program that simulates CPU process scheduling. The scheduler must support five different scheduling algorithms, each with its own rules for determining which process occupies the CPU at any given clock cycle.

CPU scheduling is a fundamental operating system concept that determines the order in which processes access the CPU. Different scheduling strategies trade off between fairness, throughput, response time, and complexity. In this simulation:

- Each scheduling algorithm reads a list of processes with their associated parameters.
- The scheduler produces a clock-by-clock execution trace, showing which process runs (or if the CPU is idle) at each time unit.
- Tie-breaking rules ensure deterministic output.

2 Input and Output Format

2.1 Input Format

Your program will process a single input line. The first token identifies the scheduling algorithm, followed by process descriptors and (for Round Robin) an optional quantum value.

<Algorithm> <Process1> <Process2> ... [<Quantum>]

Each process descriptor is a hyphen-separated string whose fields depend on the algorithm. The exact format for each algorithm is described in Section 3.

2.2 Output Format

Your output should be a single line representing the CPU execution timeline from time 0 until all processes have completed. Each character in the output corresponds to one clock cycle:

- A process ID letter (e.g., A, B) indicates the process running during that cycle.
- X indicates an idle CPU cycle (no process available to run).

<execution timeline string>

For example, XAAABB means the CPU was idle for 1 cycle, then process A ran for 3 cycles, then process B ran for 2 cycles.

3 Scheduling Algorithms

3.1 First Come First Serve (FCFS)

Processes are executed in the order of their arrival time. Once a process starts, it runs to completion without preemption.

Input format:

FCFS <ID>-<BurstTime>-<ArrivalTime> ...

- ID: single uppercase letter identifying the process.
- BurstTime: total CPU time units the process requires.
- ArrivalTime: clock cycle at which the process becomes available.

Example:

FCFS A-3-1 B-2-2

Output:

XAAABB

Explanation: The CPU is idle at cycle 0 (no process has arrived). Process A arrives at cycle 1 and runs for 3 cycles. Process B arrives at cycle 2 but must wait until A finishes, then runs for 2 cycles.

Tie-breaking: If two processes arrive at the same time, the one appearing first in the input is scheduled first.

3.2 Shortest Job First (SJF)

All processes are assumed to arrive at time 0. The process with the shortest burst time is selected first. SJF is non-preemptive: once started, a process runs to completion.

Input format:

SJF <ID>-<BurstTime> ...

Example:

SJF A-2 B-1 C-4

Output:

BAACCCC

Explanation: B (burst 1) runs first, then A (burst 2), then C (burst 4).

Tie-breaking: If two processes have the same burst time, the one appearing first in the input is scheduled first.

3.3 Shortest Remaining Time First (SRTF)

SRTF is the preemptive version of SJF. At every clock cycle, the scheduler picks the ready process with the smallest remaining burst time. A running process is preempted if a newly arrived process has a shorter remaining time.

Input format:

SRTF <ID>-<BurstTime>-<ArrivalTime> ...

Example:

SRTF A-4-0 B-2-1 C-5-0

Output:

ABBAAACCCCC

Explanation: At cycle 0, only A (remaining: 4) and C (remaining: 5) are available; A runs. At cycle 1, B arrives (remaining: 2), preempting A (remaining: 3). B runs for 2 cycles and completes. A resumes and finishes its remaining 3 cycles. Then C runs for 5 cycles.

Tie-breaking: If two processes have equal remaining time, the one appearing first in the input is scheduled first.

3.4 Priority First (PF)

Processes are scheduled based on a priority value. A lower priority number indicates higher priority. PF is preemptive: a higher-priority process that arrives will immediately

preempt the currently running process.

Input format:

PF <ID>-<BurstTime>-<ArrivalTime>-<Priority> ...

- Priority: a non-negative integer. Lower value = higher priority.

Example:

PF A-2-0-2 B-5-1-1

Output:

ABBBBBA

Explanation: At cycle 0, only A (priority 2) is available and starts running. At cycle 1, B (priority 1) arrives; since B has higher priority, it preempts A. B runs for 5 cycles. After B finishes, A completes its remaining 1 cycle.

Tie-breaking rules (applied in order):

1. If two processes share the same priority, choose the one with the shorter remaining time (SRTF behavior).
2. If both priority and remaining time are equal, choose the one appearing first in the input.

3.5 Round Robin (RR)

Processes are executed in a cyclic order. Each process may run for at most Q consecutive cycles (one quantum) before being preempted and moved to the back of the ready queue. All processes arrive at time 0 and are enqueued in input order.

Input format:

RR <ID>-<BurstTime> ... <Quantum>

- ID: single uppercase letter identifying the process.
- BurstTime: total CPU cycles the process requires.
- Quantum: a positive integer Q given as the last token in the input line.

Scheduling rules:

1. At time 0, enqueue all processes in the order they appear in the input.
2. Dequeue the front process and allocate a full slot of Q cycles.

3. Output the process ID for each cycle it actively runs (up to its remaining time).
4. If the process finishes before the quantum expires, output X for each remaining cycle in that quantum slot. The CPU does not switch to the next process mid-quantum.
5. If the process still has remaining time after the full quantum, re-enqueue it at the back of the ready queue.
6. If the process finishes within its quantum (with or without idle padding), discard it.
7. If the queue is ever empty while cycles remain, output X for each idle cycle.
8. Repeat from step 2 until all processes are complete.

Example:

RR A-3 B-4 C-2 2

Output:

AABBCCAXBB

Step-by-step trace (quantum Q = 2):

Turn | Process | Remaining Before | Cycles Run | Idle Padding | Output | Remaining After
1    | A       | 3                | 2          | 0            | AA     | 1 (re-enqueued)
2    | B       | 4                | 2          | 0            | BB     | 2 (re-enqueued)
3    | C       | 2                | 2          | 0            | CC     | 0 (done)
4    | A       | 1                | 1          | 1            | AX     | 0 (done)
5    | B       | 2                | 2          | 0            | BB     | 0 (done)

Explanation: At turn 4, process A only needs 1 cycle but holds the full quantum of 2. It outputs A for its active cycle and X for the unused remainder. The next process B only starts after the full quantum slot ends.

Tie-breaking: When multiple processes are re-enqueued at the same time step, they join the back of the queue in their original input order.

7 Assumptions

- Input tokens are strictly separated by a single space.
- The number of processes will not exceed 10.
- Process IDs are unique single uppercase letters.
- Burst times and arrival times are non-negative integers.
- Priority values are non-negative integers (PF only).
- The quantum (RR only) is a positive integer and is always the last token.
- The input is otherwise well-formed; no invalid tokens will appear.
- If two or more processes have equal selection criteria, apply tie-breaking rules as described per algorithm.
- Output is a single line with no trailing spaces or newline beyond the terminating newline character.
- The output line will not exceed 1024 characters.
